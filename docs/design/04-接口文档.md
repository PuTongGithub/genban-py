# 对外接口文档

## 基础信息

- **Base URL**: `http://localhost:8000`
- **接口前缀**: `/api`

---

## 1. 用户登录

### 接口描述
用户登录，获取访问令牌

### 请求信息

**方法**: `POST`  
**路径**: `/api/login`  
**Content-Type**: `application/json`

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| user_id | `string` | 是 | 用户ID |
| password | `string` | 是 | 密码 |

### 请求示例

```bash
curl -X POST "http://localhost:8000/api/login" \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": "alice",
    "password": "123456"
  }'
```

### 响应示例

**成功**:
```json
{
  "token": "abcd1234...",
  "expires_at": 1700000000,
  "error": ""
}
```

**失败**:
```json
{
  "token": "",
  "expires_at": 0,
  "error": "User not found"
}
```

### 响应字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| token | `string` | 访问令牌，有效期7天 |
| expires_at | `integer` | 过期时间（秒级时间戳） |
| error | `string` | 错误信息，成功为空 |

## 2. 提交用户输入

### 接口描述
提交用户输入到对话系统，返回该条消息的 chatId。

### 请求信息

**方法**: `POST`  
**路径**: `/api/submit`  
**Content-Type**: `application/json`  
**认证**: `Authorization: Bearer <token>`

### 请求头

| 头信息 | 必填 | 说明 |
|--------|------|------|
| Authorization | 是 | Bearer Token，格式：`Bearer <token>` |

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| user_input | `string` | 是 | 用户输入内容 |

### 请求示例

```bash
curl -X POST "http://localhost:8000/api/submit" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer abcd1234..." \
  -d '{
    "user_input": "你好，请介绍一下自己"
  }'
```

### 响应示例

**成功**:
```json
{
  "chat_id": "msg_001"
}
```

**失败**:
```json
{
  "detail": "Unauthorized"
}
```

### 响应字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| chat_id | `string` | 用户消息的唯一标识 |

## 3. 流式对话（SSE）

### 接口描述
建立 SSE 长连接，服务端主动推送对话过程中的所有消息。

**使用流程**：
1. 先调用 `/api/submit` 提交用户输入
2. 再调用 `/api/stream` 建立长连接接收消息

**特点**：
- 长连接持续保持，不会主动关闭
- 助手会主动推送消息（包括工具调用、多轮对话等）
- 前端需要持续监听并处理新消息

### 请求信息

**方法**: `POST`  
**路径**: `/api/stream`  
**认证**: `Authorization: Bearer <token>`

### 请求头

| 头信息 | 必填 | 说明 |
|--------|------|------|
| Authorization | 是 | Bearer Token，格式：`Bearer <token>` |

### 请求参数

无

### 请求示例

```bash
curl -X POST "http://localhost:8000/api/stream" \
  -H "Authorization: Bearer abcd1234..."
```

### 响应格式

**Content-Type**: `text/event-stream`

响应采用 SSE (Server-Sent Events) 格式，包含以下事件类型：

#### 1. message 事件

推送对话过程中的每条消息，包含用户消息、助手消息、工具调用消息等。

**事件格式**:
```
event: message
data: {"type": "assistant", "id": "msg_xxx", "role": "assistant", "content": "回答内容", "reasoning_content": "思考过程", "tool_calls": null}
```

**响应字段说明**:

| 字段 | 类型 | 说明 | 适用角色 |
|------|------|------|----------|
| type | `string` | 消息类型，可能为 `user`, `assistant`, `tool`, `command`, `error` | 所有 |
| id | `string` | 消息唯一标识（chatId） | 所有 |
| role | `string` | 消息角色，可能为 `user`, `assistant`, `tool` | 所有 |
| content | `string` | 消息内容 | 所有 |
| reasoning_content | `string` | AI 的思考过程（仅 assistant 角色） | 仅 `assistant` |
| tool_calls | `list` | 工具调用信息 | 仅 `assistant` |

**消息类型说明**:

| 类型 | 说明 |
|------|------|
| `user` | 用户输入的消息 |
| `assistant` | AI 助手的回复消息 |
| `tool` | 工具调用的返回结果 |
| `command` | 指令执行结果（如切换模型、开启思考等） |
| `error` | 系统错误消息，当对话处理过程中发生异常时返回，用于向用户告知错误信息 |

#### 2. complete 事件

当单轮对话完成时触发。注意：连接不会关闭，会继续接收后续消息。

**事件格式**:
```
event: complete
data: {}
```

#### 3. error 事件

当发生错误时触发。

**事件格式**:
```
event: error
data: {"error": "错误描述信息"}
```

### 响应示例

#### 示例 1：普通对话（无工具调用）

```
event: message
data: {"type": "user", "id": "msg_001", "role": "user", "content": "你好", "reasoning_content": "", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "", "reasoning_content": "让我思考", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "", "reasoning_content": "让我思考一下...", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "你好", "reasoning_content": "让我思考一下...", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "你好！我是你的 AI 助手", "reasoning_content": "让我思考一下...", "tool_calls": null}

event: complete
data: {}

# 连接保持，等待下一条用户输入...
```

#### 示例 2：复杂对话（包含工具调用）

```
event: message
data: {"type": "user", "id": "msg_001", "role": "user", "content": "查询北京天气", "reasoning_content": "", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "", "reasoning_content": "用户想查询天气", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_002", "role": "assistant", "content": "我来为您查询天气", "reasoning_content": "用户想查询天气，我需要调用天气工具", "tool_calls": [{"function": {"name": "get_weather", "arguments": "{\"location\": \"北京\"}"}}]}

event: message
data: {"type": "tool", "id": "msg_003", "role": "tool", "content": "北京今天晴，25°C", "reasoning_content": "", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_004", "role": "assistant", "content": "", "reasoning_content": "工具返回了天气信息", "tool_calls": null}

event: message
data: {"type": "assistant", "id": "msg_004", "role": "assistant", "content": "北京今天天气晴朗，气温 25°C", "reasoning_content": "工具返回了天气信息", "tool_calls": null}

event: complete
data: {}

# 连接保持，等待下一条用户输入...
```

#### 示例 3：系统错误

当对话处理过程中发生异常时，会推送 `error` 类型的消息向用户告知错误。

```
event: message
data: {"type": "user", "id": "msg_001", "role": "user", "content": "你好", "reasoning_content": "", "tool_calls": null}

event: message
data: {"type": "error", "id": "msg_002", "role": "assistant", "content": "由于本次执行内容过长，我无法处理。请重新输入。", "reasoning_content": "", "tool_calls": null}

event: complete
data: {}

# 连接保持，等待下一条用户输入...
```

### 注意事项

1. **认证**: 需要在请求头中携带 `Authorization: Bearer <token>`
2. **Token 获取**: 通过 `/login` 接口获取 token，有效期 7 天
3. **消息去重**: 前端应基于 `id` 字段去重，避免重复显示，尤其注意不要重复展示用户输入的消息
4. **错误处理**: 需要监听 `error` 事件处理异常情况，同时注意 401 认证失败响应
5. **连接管理**: 前端应在页面关闭或用户登出时主动关闭连接
